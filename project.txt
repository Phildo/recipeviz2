PROJECT: Recipe Visualizer

STACK:
- Docker
- Nginx
- PostgreSQL
- Vanilla HTML/CSS/JS
- python server
- Anthropic LLM API

GOALS:
- Data-driven recipe visualization website
- User can submit a recipe as: a url, a block of text, or a series of images
- The server will take one pass on the input data, sending it to the latest sonnet model to extract the recipe in markdown form (and stripping non-recipe information)
- The server will take that reduced recipe text and take a second pass, this time sending it to the latest opus model, to extract its information according to the schemas of our database (in json form)
- The schema used to describe recipes is non-standard. The goal is to visualize the flow of ingredients from their base forms, through processes that modify them, across tools, combining into the final result.
- At a high level, the schema has:
  - recipes (id, name, source type, high level recipe metadata)
  - ingredients (id, name) //used across recipes
  - tools (id, name) //used across recipes
  - actions (id, name) //used across recipes
  - units (id, name) //used across recipes
  - recipe_transforms (recipe_id, id, action_id, active, name, description, duration_minutes) //active is false when the transform is accomplished passively (e.g. "wait")
  - recipe_transform_io (recipe_id, id, recipe_transform_id, is_output, ingredient_id, tool_id, pipe_uid, display_name, color, unit_id, amount) //can be ingredient or tool, with some unique pipe ID to keep track of where it came from/is going
  - recipe_source_distillations (recipe_id, id, distilled_text)
  (there may be more tables, or more mebers to any of these tables)
- Recipes will be visualized in browser as pipes flowing into and out of transforms (pipes will be matched up using pipe_uid), tool pipes will not be shown, but will be used to ensure dependency ordering
- When new ingredient or tool or action or unit names are encountered in the AI response, they are merged into the global tables (eventually each will be associated w/ a generated image for reuse)
- Some example transforms:

- note that transforms must be higher fidelity than described in plain language; every step must be accounted for so it can be represented as a graph.
- The web page will automatically consolidate all needed base ingredients (ingredient inputs to transforms that are NOT generated by dependee transforms) and display to user
- It will also display the quantity and units inline with each step

- There are no additional software dependencies beyond core stack
- Use direct database queries, minimal abstraction
